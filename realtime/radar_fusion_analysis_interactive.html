Interactive analysis and commands console for target tracking nodes

Functional Summary

Both files are ROS 2 nodes designed to synchronize and fuse YOLO image bounding boxes with range/azimuth data provided by forward-facing radar sensors. The scripts project polar radar returns onto the 2D camera coordinate space to execute late geometric association.

Key Sensors
RGB-D + Radar
RealSense depth maps provide localized confirmation while radar scans provide accurate ranges & velocities.
Fusion Strategy
Geometric Late
Evaluated at the bounding-box level. Projection relies on camera focal length and center point intrinsics.
Primary Targets
Person & Car
The advanced debug node isolates humans and vehicles to optimize detection confidence on CPU.

First-Run Model Selection

The CPU debug script now starts with an ONNX-first policy. This reduces first-run surprises on the test machine while still keeping a PT fallback.

01
Look for ONNX
Default model path is realtime/yolo12n.onnx. If it exists, the node loads it directly.
02
Try PT Export
If ONNX is missing but yolo12n.pt exists, Ultralytics tries to export an ONNX file using the configured image size.
03
Use Exported ONNX
When export succeeds, the newly created ONNX file is used for inference instead of the PT model.
04
Fallback to PT
If export fails, the script logs a warning and falls back to yolo12n.pt so the node can still run.
Recommended test setup: keep both realtime/yolo12n.onnx and realtime/yolo12n.pt beside the script. ONNX runs first; PT is only backup.

Live Runtime Pipeline

This is the actual data flow in radartracks_processing_person_car_cpu_debug.py. Camera frames drive the callback; radar is matched from the nearest buffered scan.

01
Sync RGB + Depth
Color and aligned depth images are synchronized with queue size 30 and slop 0.1s.
02
YOLO Throttle
YOLO runs at max_yolo_hz, default 5Hz. Between detections, the latest boxes are reused.
03
Closest Radar Scan
Radar scans are buffered. The closest scan to the camera timestamp is accepted only within 0.15s.
04
Gate, Track, Publish
The node projects radar to image space, gates against boxes/depth, updates tracks, then publishes /detections.
Camera callback frames
Runs whenever synchronized color + depth arrive.
YOLO detections
Runs slower; cached boxes are marked fresh until 0.6s.
Radar scans
Fast scans are buffered; only nearest-time scan is fused.

Runtime Notes

OpenCV window title

Fusion Pipeline is only the window title. It does not change ROS topics or output data. If the machine has no GUI, display is disabled and publishing continues.

Camera FPS vs detection FPS

The debug window shows continuous camera frames. YOLO boxes update only when YOLO runs; otherwise cached boxes are shown with a cache-age note.

No radar input

Camera and YOLO visualization can still run. Fusion has zero matches and the node does not publish fused detections.

No camera or no depth input

The synchronized callback does not run. Radar may buffer, but there is no YOLO, no projection, no fusion, and no /detections output.

Fusion Decision Walkthrough

Think of fusion as one question: does this radar point belong to this YOLO box? For every camera frame, the script compares YOLO boxes, depth-camera distance, and radar returns, then accepts a match only when the signals agree enough.

1. YOLO boxes

YOLO says what object is visible and where it is in the image.

car box: x = 300..500
2. Depth value

The depth image says approximately how far the object is from the camera.

box depth = 12 m
3. Radar returns

Radar says something exists at a range, angle, reflection strength, and relative speed.

range = 13 m, RCS = -12
Step 1: Radar Projection

Radar gives range and azimuth. The script projects that radar return into the camera image as a pixel coordinate u. 

range = 13 m
azimuth = angle left/right
projected pixel u = 410

YOLO box x = 300..500
Is u = 410 inside the box? yes

If the projected radar pixel is horizontally inside or near the YOLO box, the radar return becomes a possible match.

Step 2: RCS Gate

RCS means radar reflection strength. High RCS is a strong echo. Low RCS is more likely weak noise or a ghost return. 

min_rcs = -25

RCS -10 -> keep
RCS -22 -> keep
RCS -32 -> reject

Increase min_rcs to reject more weak radar points. Decrease it to keep more weak points, which is more sensitive but creates more false matches.

Step 3: Depth Gate

Depth camera and radar range should roughly agree. The new script uses an adaptive tolerance, because far objects need more tolerance than close objects. 

depth_gate = max(depth_gate_base_m,
                 depth_gate_ratio * radar_range)

default = max(1.6 m, 0.16 * radar_range)

At 13 m:
0.16 * 13 = 2.08 m
depth_gate = 2.08 m
YOLO depth = 12 m
Radar range = 13 m
Difference = 1 m

1.0 m -> accepted
4.0 m -> rejected
Step 4: Score

If a radar point passes the gates, the script gives each possible radar-box pair a score. The best one-to-one pair wins. 

score =
  0.48 * horizontal_score +
  0.32 * depth_score +
  0.15 * yolo_confidence +
  0.05 * rcs_score
  • Horizontal alignment is most important.
  • Depth agreement is second.
  • YOLO confidence helps the decision.
  • RCS helps a little, but does not dominate.
Simple example Value Decision
YOLO sees car box x = 300..500, depth = 12 m, confidence = 0.80 Candidate object exists in the camera image.
Radar return range = 13 m, projected u = 410, RCS = -12 Radar point is near the same horizontal image area.
Gate checks u inside box: yes, RCS: yes, depth diff = 1 m: yes fusion accepted
Output class_name, range, x/y position, track_id, vrel_x, vrel_y The object is tracked over time with fused position and estimated velocity.

Short version: YOLO tells what object is visible and where it is in the image. Depth tells camera distance. Radar gives range, angle, speed, and reflection strength. Fusion accepts the match only when those signals agree enough.

Fusion Mechanics & Gating Theory

Detailed breakdown of the projection math, association scoring, and CPU optimization techniques implemented in the new debug script.

1. Fusion Construction

Polar to Camera coordinates: Converts radial range (R) and azimuth (θ) to Cartesian coordinates (xforward = R cos θ, yleft = R sin θ).

2D Projection: Projects 3D camera coordinates to image coordinates (u, v) using the focal length fx and optical center cx: u = (fx * yleft) / xforward + cx,   v = cy The vertical v is fixed at the horizon (cy) because standard 2D radar lacks height/elevation metrics.

Robust Depth confirmation: Extracts a median depth value within a window around the bounding box center to avoid camera noise/holes.

2. Why the CPU Debug Node is Better
  • YOLO Rate Throttling: Caps YOLO at e.g., 5 Hz and reuses cached detections. Cached boxes are used for fusion only while they are fresh.
  • Multi-Criteria Association: Replaces simple collision checking with a weighted matching score: Score = 0.48*Horizontal + 0.32*Depth + 0.15*Conf + 0.05*RCS It then performs greedy 1-to-1 matching to prevent double-association.
  • Dynamic Depth Gating: Scales depth limits with distance: max(1.6m, 0.16 * range). This matches the physical reality of sensor uncertainty scaling at distance.
  • Stateful Temporal Smoothing: Maintains continuous tracking IDs using spatial proximity. Smoothes position updates using an Exponential Moving Average filter to suppress measurement jitter.

Main status numbers

boxes: 3 means YOLO found three target objects. radar returns: 11 means the fake radar generated eleven returns, including clutter. matches: 3 means three radar returns were accepted as belonging to YOLO boxes.

Fusion label format

#1 car R:34.3 C:50% S:0.69 means track ID 1, class car, fused range 34.3 m, camera contributes 50 percent of the fused position, and association score is 0.69.

Offline fake mode

The bottom label OFFLINE TEST: fake YOLO depth + fake radar means only the camera video is real. Depth and radar are synthetic, so this verifies visualization, timing, projection, gating, and tracking logic without ROS hardware.

When something looks wrong

Boxes with no red match usually means radar projection or depth gates are too strict. Red points far from the visible object suggest camera FOV, intrinsics, or lateral radar offset needs tuning.

Node Parameter Reference

Key ROS 2 parameters defined in radartracks_processing_person_car_cpu_debug.py to customize node behavior:

  • model_path (default: realtime/yolo12n.onnx): ONNX is tried first. If missing, the script tries to export from PT, then falls back to PT.
  • yolo_imgsz (default: 480): BGR input resolution for the YOLO neural network model.
  • max_yolo_hz (default: 5.0): Maximum YOLO inference rate. Camera frames can still update faster than detections.
  • max_cached_yolo_age_sec (default: 0.6): Cached boxes older than this are displayed but not used for fusion.
  • radar_match_slop_sec (default: 0.15): Maximum allowed time gap between camera frame timestamp and closest radar scan.
  • torch_num_threads (default: min(4, CPU count)): Limits CPU cores assigned to PyTorch calculations.
  • min_rcs (default: -25.0): Rejects noise returns showing low Radar Cross Section reflectivity.
  • depth_gate_base_m (default: 1.6) & depth_gate_ratio (default: 0.16): Configures spatial matching windows.
  • allow_missing_depth (default: True): Continues to attempt projection/association when camera depth values are missing or zero.
  • debug_visuals (default: True): Enables OpenCV windows. If display fails, the node disables visuals and continues running.

What These Scripts Are

Both files are ROS 2 nodes that combine YOLO camera detections, RealSense aligned depth, and radar returns or tracks into fused object detections.

Expected performance, not benchmarked
YOLO model requested
yolo12n.pt
This means the scripts ask Ultralytics for a YOLO12 nano weight file. The workspace does not include the weight file.
Fusion style
Geometric late fusion
YOLO supplies class and box. Radar supplies range, azimuth, RCS, and velocity.
Tracking level
Limited
The scripts are not Kalman filters. The all-classes file consumes radar tracks, but it does not estimate tracks itself.

Core idea

The camera sees object appearance. YOLO detects boxes and classes such as person or car. The radar measures physical distance, bearing, reflection strength, and relative velocity. The script projects radar into the image and accepts a fusion when the projected radar point aligns with a YOLO box.

The important assumption The radar and camera are treated as if they are almost perfectly aligned. There is no full radar-to-camera extrinsic calibration matrix in these scripts.

Local environment facts

The code imports from ultralytics import YOLO and constructs YOLO("yolo12n.pt"). The repository has no .pt file and no dependency lock file. In the checked local Python environment, ultralytics and torch were not installed, so the exact installed package version cannot be inferred from this workspace.

Mathematical projection

The radar point is converted from polar coordinates to a simple camera-frame approximation.

x_r = range * cos(azimuth)   # forward
y_r = range * sin(azimuth)   # left/right

pt_cam = [y_r, 0, x_r]

u = fx * pt_cam[0] / pt_cam[2] + cx
v = fy * pt_cam[1] / pt_cam[2] + cy

Why vertical matching is fragile

Because pt_cam[1] is always zero, v is effectively always cy. Radar has no image-height measurement here. The lean script avoids depending on vertical radar position by using horizontal overlap and depth consistency.

Concrete Next Steps

Use this order on a normal CPU PC: first prove the sensor fusion works with the PyTorch model, then export ONNX, then compare speed and accuracy.

Recommended path

Run order

Confirm ROS topics are alive

Check color, aligned depth, camera info, and radar scan before running YOLO. If one topic is missing, fusion cannot be trusted.

Run the CPU debug node first

Use the new person/car script. Keep debug windows enabled until you see YOLO-only, radar-only, and fusion behavior separately.

Tune CPU cost before tuning fusion

Start with yolo_imgsz=480 and max_yolo_hz=5. If slow, reduce to 416 and 3 Hz.

Tune projection and gates

Adjust lateral offset, RCS gate, and depth gate only after the separate debug views show that YOLO and radar are individually correct.

Export ONNX only after baseline works

ONNX is for speed and deployment. It should not be the first debugging step because it adds another dependency and runtime layer.

Exact commands

# 1. Check topics
ros2 topic list
ros2 topic hz /camera/camera/color/image_raw
ros2 topic hz /camera/camera/aligned_depth_to_color/image_raw
ros2 topic hz /radar_scan
# 2. CPU baseline run
python realtime/radartracks_processing_person_car_cpu_debug.py --ros-args \
  -p device:=cpu \
  -p yolo_imgsz:=480 \
  -p max_yolo_hz:=5.0 \
  -p torch_num_threads:=4
# 3. If CPU is too slow
python realtime/radartracks_processing_person_car_cpu_debug.py --ros-args \
  -p device:=cpu \
  -p yolo_imgsz:=416 \
  -p max_yolo_hz:=3.0 \
  -p show_radar_debug:=false
# 4. Export ONNX after baseline is correct
yolo export model=yolo12n.pt format=onnx imgsz=480 simplify=True
# 5. Test ONNX in the same node
python realtime/radartracks_processing_person_car_cpu_debug.py --ros-args \
  -p model_path:=yolo12n.onnx \
  -p device:=cpu \
  -p yolo_imgsz:=480 \
  -p max_yolo_hz:=5.0

Tuning priorities

CPU too slow
Lower yolo_imgsz, lower max_yolo_hz, disable extra debug windows, then try ONNX.
YOLO misses person/car
Raise yolo_imgsz to 640 or lower yolo_conf from 0.35 to 0.25.
Radar dots shifted left/right
Tune radar_lateral_offset_m. Positive and negative signs must be tested with a known target in front of the car.
Radar has many false returns
Increase min_rcs, reduce max_azimuth_deg, and watch the radar-only debug view before changing fusion logic.
Fusion matches wrong object
Tighten depth_gate_base_m or depth_gate_ratio, and verify RealSense depth is not zero at the box center.

Problem solver

ONNX decision

Try ONNX if the PyTorch .pt version already detects correctly but CPU inference time is too high. Keep the same imgsz when comparing. Accept ONNX only if it is faster and class boxes look the same.

# Accept ONNX when:
# - YOLO boxes are the same or close
# - fusion match count is not worse
# - inference ms is lower in the YOLO debug HUD
